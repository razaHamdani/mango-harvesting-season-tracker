# PLAN: Runtime test harness for AamDaata

## Goal

Add an integration test suite that exercises the three recent fixes
end-to-end against a real Postgres + Supabase Auth + Storage stack.

In-scope scenarios:

1. **RC-2 (payment TOCTOU)** — concurrent double-record must yield exactly
   one successful payment; the other call returns "already recorded".
2. **RC-1 (atomic season creation)** — `createSeason` with installments
   that do not sum to `predetermined_amount` must fail *and leave no rows
   behind* in `seasons`, `season_farms`, or `installments`.
3. **PHOTO-1 (client-side upload + server-side path validation)** —
   (a) an uploaded path within the user's namespace persists on the
   record; (b) a spoofed path outside the namespace is silently dropped
   to `null`.

Not in scope: Playwright/browser UI tests, load tests, visual regression.

## Tech stack

- **Runner:** Vitest
- **DB/Auth/Storage:** Supabase local dev via `npx supabase start`
  (Docker-backed). Config lives in `supabase/config.toml` + migration
  files under `supabase/migrations/`.
- **Schema delivery:** split the existing monolithic `supabase/schema.sql`
  into one migration `supabase/migrations/0001_initial_schema.sql` so
  `supabase db reset` + `supabase start` both apply it.
- **Server action invocation:** import the actions directly; replace
  `@/lib/supabase/server` `createClient` via `vi.mock` with a factory
  that returns an authed `@supabase/supabase-js` client. Replace
  `next/cache` `revalidatePath` with a no-op.

## Architecture

```
aam-daata/
  vitest.config.ts
  supabase/
    config.toml              (generated by `supabase init`)
    migrations/
      0001_initial_schema.sql  (moved from schema.sql)
    schema.sql               (kept as the canonical readable copy)
  tests/
    setup.ts                 vi.mock for server client & next/cache
    helpers/
      env.ts                 loads SUPABASE_URL / keys from supabase start output
      admin.ts               service-role client for seeding / truncate
      user.ts                createTestUser() -> authed client + userId
      fixtures/
        test.jpg             1x1 px JPEG
    payments.test.ts         RC-2: concurrent double-record
    seasons.test.ts          RC-1: bad sums, atomic rollback
    photos.test.ts           PHOTO-1: namespace validation
  .env.test                  SUPABASE_URL, anon + service keys (gitignored)
```

## Test flow

- `beforeAll`: read local Supabase URL/keys from env (populated by
  `supabase status`), create an admin client, ensure schema is applied.
- `beforeEach`: truncate application tables (not `auth.users` or
  `storage.objects`; we keep the test user across tests for speed) using
  the service-role client.
- Each test creates the data it needs (farm, season, installment) and
  exercises exactly one scenario.
- `afterAll`: leave the local stack running — devs can re-run tests
  without incurring the cold-start cost again.

## How to run

```bash
npx supabase start           # one-time per session (or per reboot)
npm run test                 # vitest
npx supabase stop            # when done, to free containers
```

`npm run test` will be added to `package.json`.

## Risks / deferred

- **Storage RLS:** tests write to the `aam-daata-photos` bucket. Local
  Supabase starts with a permissive default. If/when we add a strict
  INSERT policy scoped by `auth.uid()`, that policy goes in the initial
  migration so tests exercise it.
- **Type drift on the RPC:** `create_season_with_children` still needs
  `supabase gen types` regen after the first `db reset`. The `as any`
  cast in `season-actions.ts` stays until then.
- **Windows path quirks:** Supabase CLI + Docker Desktop on Windows
  occasionally stalls on first pull. First `supabase start` may take
  several minutes; subsequent starts are fast.

## Immutability

This plan is the contract for this feature. Scope changes (adding a
4th scenario, adopting Playwright) require a new PLAN with a supersede
note rather than editing this one.
